    Key Takeaways

    1. Custom Tools are worth it.
    2. Borrow, don’t build.
    3. Understand the Relationship between your engine and your assets.
    4. Prioritize on necessary features first.

    Focus on Modular Design

    Our production timeline focused around weekly playtesting and rapid iteration. To enable quick additions and changes to the game, our codebase had to be extremely modular and easily tweakable. With this idea in mind, our codebase centered around “data-driven design”, which is outlined in Jonas Tyroller’s video Best Code Architecture for Indie Games.

    Data-driven design emphasizes modularity through rigid segmentation. One of the largest benefits of this is being able to store all design-related parameters in separate resource files, which are read by the game at runtime to determine behaviors. This would allow the whole game to be easily tweakable without even opening the Godot editor, which would be very designer-friendly.

    Data-driven design also emphasizes modularity by creating systems that can be applied to many different contexts, like creating a “hurt” component that could be applied to anything to make it capable of having health and taking damage. Since platformers can have many different systems working with each other in creative ways, using modular components would make it easy to tweak design and smoothly enable complex interactions.

    Using Data Driven Design Patterns

    Throughout development, data-driven design proved to be very useful, especially as our game’s design work became a larger focus. Having numerical parameters in separate text files made it extremely easy to modify how entities behaved. This was especially helpful when tweaking the Player, which had to undergo many design changes throughout development. Similarly, prioritizing modular features helped our systems scale when we wanted to add or change features.

    Investing into a debug menu also paid off; it allowed us to see how values like Player velocity & states changed in real-time, which expedited debugging. Being able to swap from scene-to-scene without having to close the game and navigate the editor was also a great time-saver.

    Plugins

    Plugins proved to be very helpful & saved us a lot of time. LimboAI simplified the creation of state machines for the Player and Ice Scream enemies, and was very easy to work with.

    We originally planned to use InkGD to handle dialogue sequences, but found its ease of use lacking. With that in mind, we switched to DialogueManager early on in our production. DialogueManager worked right out-of-the-box and streamlined our dialogue implementation. Unfortunately, the plugin had some shortcomings we had to wrestle with, like dialogue scripts only being able to call Autoloads and not having built-in compatibility with dialogue interruptions.

    We started using PhantomCamera midway through development and found it to be a very powerful tool with multiple convenient and intuitive features like camera priorities. However, we encountered issues with understanding the more complex features of the tool. This was largely because the plugin was relatively new, and thus did not have the ease of use associated with more mature tools. Additionally, the plugin’s documentation was broad but not very detailed, leading to times when we struggled to understand PhantomCamera’s full feature set.

    Godot & GDScript

    Godot proved to be very powerful for developing our game, but served as a double-edged sword. The engine is very flexible, providing many options to accomplish the same task while also making it easy to rebuild a system if one option does not work out. Documentation is also plentiful and detailed, which helped us understand the engine’s capabilities quickly.

    However, Godot and GDScript were not as mature as other engines & languages, which created some roadblocks when Godot and GDScript lacked useful features, like structs and interfaces. We also encountered issues using Godot’s JSON resources, which would not open up in the editor and forced us to use an external text editor for parameter tweaks. The sheer number of options to create systems also made it difficult to know which option would work best for the game. During development, we under-utilized TileMapLayers and over-used CanvasLayers, both of which forced us to refactor our architecture later in the project.

    Architecture & YAGNI

    A major issue we struggled with throughout this project was creating systems that accomplished only what we wanted, and nothing more or less. As a team of less experienced programmers, we found it hard to know in advance exactly what functionality each system would need. This was further exacerbated by our constantly shifting design. This resulted in times when we over-engineered or under-engineered a system, both of which detracted from our development time. We had multiple occurrences where we succeeded and failed to create properly-scoped systems.

    Lesson 1: Custom Tools

    We developed a rudimentary debug menu early on in development, sort of on a whim. It only had two features: a debug property panel that let us watch added variables change during runtime and a noclip panel that let us teleport the player to any room in the game. It seems somewhat insignificant at first glance, but the debug panel singlehandedly enabled us to iterate on room designs super quickly. Developing custom features that players will never see may seem like a waste of time for such a short timeframe but it actually paid off massively in the end.

    Lesson 2: Borrow, don't build

    There are several features in Leaves of Fate that we simply would not have had the time to add if we developed them ourselves. The dialogue system, smooth camera manipulations, and player states all depend heavily on Godot plugins we found and implemented into our tech stack. These three features are fundamental to the gameplay experience and would not be nearly as polished if we had tried to build those systems ourselves instead of borrowing from existing projects.

    Lesson 3: Relationship between Engine and Assets

    As a technical team member, it is important that you understand how the engine processes art to properly communicate to your art team how to make assets. This came about specifically with how Godot handles importing of art vs how it tiles art. There are things that the engine did under the hood that shifted line weights that were unknown until assets were already mostly done.

    Lesson 4: Prioritize Necessary Systems

    We sometimes fell into the trap of designing code systems for things we thought we would need instead of what we actually needed at the moment. Learning the difference between wanting a feature and needing a feature is important because working on something we never use wastes our already limited time.
